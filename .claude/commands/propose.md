Write a design proposal for the described change or feature. Do not write any code or edit any files.

The proposal should include:

1. **Context** — what problem is being solved and why it matters now
2. **Constraints** — what must be true about the solution (settled decisions, invariants, external requirements)
3. **Options** — 2–3 realistic alternatives, each with trade-offs. Do not manufacture options — only include ones that are genuinely viable given the constraints.
4. **Recommendation** — which option and why. Be direct. If one option is clearly better, say so.
5. **Open questions** — what needs a human decision before implementation can start
6. **Files that will change** — list the files and a one-line description of what changes in each

Keep proposals concise. The goal is alignment before implementation, not a design document that will never be read again. A proposal that fits in a single screen is better than one that requires scrolling.

After writing the proposal, ask: "Ready to implement, or do you want to discuss any part of this first?"
